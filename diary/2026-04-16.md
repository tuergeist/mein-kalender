## markAsPrivate Flag für Sync-Targets

Neues Feature: Sync-Targets können Events im Zielkalender als privat markieren. Motivation: Full Sync nutzen (Titel, Beschreibung etc. werden kopiert), aber andere Nutzer im Zielkalender sehen nur "Belegt". Umsetzung über `sensitivity` Feld auf NormalizedEvent — Outlook bekommt `sensitivity: "private"` (MS Graph API), Google bekommt `visibility: "private"`. Toggle in der Sync-Target Form. Kein opsx nötig — bekanntes Pattern wie die skip-Flags.

## Cancelled events not being deleted from DB + targets

User reported: "Christoph <> Michael" was cancelled in Outlook, but still appears in Mein Kalender UI and gets synced to target calendars as "[Sync] Abgesagt: Christoph...". The original Outlook calendar no longer shows it — only the synced copies persist.

Two issues fixed:
1. Outlook provider's delta now treats `isCancelled: true` as deletions (like Google does with `status=cancelled`)
2. Ignored events excluded from target sync — made configurable with `skipIgnored` toggle (default: on)
3. Dynamic hint in ignore dialog based on sync target config

## iCloud Calendar integration — research + implementation

Only path is CalDAV via `caldav.icloud.com` with app-specific passwords (no OAuth). Library: `tsdav` (used by Cal.com). Delta sync works via RFC 6578 sync-tokens. Full CRUD supported but updates are full-replace (ICS). Known issue: tsdav#238 service discovery problems with iCloud. No webhooks — polling only. Calendly doesn't support iCloud at all due to UX friction of app-specific passwords.

Implemented 24/28 tasks: full AppleCalendarProvider (auth, discovery, ICS parsing via ical.js, full+delta sync, CRUD via ical-generator), backend connect route, frontend onboarding page with step-by-step guide for app-specific passwords. Key decision: store Apple ID in `refreshToken` field and app-password in `accessToken` since there's no token refresh cycle. Remaining: E2E tests (need real iCloud account).

## MCP Server implementiert

Remote MCP Server als Fastify-Route auf `/api/mcp` mit Streamable HTTP Transport. Tools: list_calendars, list_events, create_event, update_event, delete_event. User-scoped — jeder Request erstellt einen MCP Server mit userId aus dem Auth-Token.

Entscheidungen:
- **Remote statt lokal**: Läuft im Deployment, nicht als stdio-Prozess. User bindet URL in Claude Desktop ein.
- **API Token statt JWT**: JWT ist kurzlebig und für MCP-Clients unpraktisch. Stattdessen `mcp_<random>` Token, SHA256-Hash in DB gespeichert. User generiert Token in Settings.
- **Stateless Transport**: Kein Session-Management — jeder Request bekommt frischen McpServer + Transport. Simpel, robust.
- **@ts-nocheck auf mcp-server.ts**: MCP SDK registerTool generics verursachen TS2589 mit Zod v3. Runtime-Validierung funktioniert korrekt.
- **Alle Write-Tools immer registriert**: readOnly-Check pro Kalender statt globalem Mode-Flag. Einfacher für Multi-Kalender-Setups.
