# 2026-04-15

## Critical sync bug found and fixed

Root cause: delta sync (Google + Outlook providers) puts ALL events тАФ including **new** events тАФ into the `updated` array when using a sync token. But `syncCalendarEntry` used `updateMany` for the `updated` path, which silently drops events that don't exist in the DB yet. New events created after the initial full sync were never persisted and never cloned to targets.

The UI showed "ok" because the source fetch succeeded тАФ there's no target sync status tracking.

Fix: changed `updateMany` to `upsert` in `sync-job.ts` for the delta `updated` path, so new events from delta responses get inserted.

Discovery: the production logs showed `84 unmapped, 0 after filter` тАФ all unmapped events were legitimately filtered (workLocation, free). The real issue was that new regular events never made it to "unmapped" because they were silently lost at the delta processing stage.
