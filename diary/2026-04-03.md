# 2026-04-03

## Cross-calendar sync debugging

Major debugging session. Target sync wasn't working at all — zero events cloned in 3 days despite 430 unmapped events.

**Discovery chain:**
1. Added diagnostic logging to cloneToTarget — found "53 unmapped, 0 after filter"
2. All 53 events were filtered out by skipFree/skipWorkLocation/skipSingleDayAllDay
3. But 59 events SHOULD have passed — the discrepancy: sourceCalendars config only included 2 calendars, one of which was another target
4. Root cause: `isTarget: false` filter in the unmapped events query excluded ALL target calendars. But Calendar (M365) is a target in one sync AND a source in another. The filter couldn't distinguish these contexts.

**Fix:** Changed from `isTarget: false` to excluding only the current target's own entry ID. This allows a calendar to be target in context A and source in context B.

**Architecture decision:** Extracted target sync into its own BullMQ queue with concurrency 1. Was running inline in every source sync job, causing concurrent cloneToTarget calls racing against each other.

**Open question:** Loop prevention currently relies on `[Sync]` title prefix + DB mappings. Cross-syncing is explicitly allowed and desired. Need to formalize the sync specification — what constitutes a loop vs legitimate cross-sync.
