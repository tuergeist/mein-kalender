## Session: Fix sync oversync / event duplication

### Key decisions and discoveries

**Root cause of duplicate [Sync] events:** The sync dedup relied solely on `TargetEventMapping` DB records — if the same real-world event appeared across multiple source calendars (e.g., personal + work Google accounts both seeing a shared meeting), each created a separate local `Event` row, and both got synced to the target. Also, if the mapping DB insert failed after the provider event was already created, the event would be re-created on next sync.

**Fix approach — fingerprint-based dedup:** Before creating target events, build a fingerprint map (`title|startTime|endTime`) from already-mapped events. Unmapped events matching an existing fingerprint just get a mapping record pointing to the existing target event — no new event created on the provider. This handles cross-calendar-entry duplication cleanly.

**Secondary fix — orphan cleanup on dup mapping failure:** If the mapping insert fails (unique constraint) after an event was already created on the target provider, the orphaned target event is now deleted immediately instead of being left as a phantom.

### Changes made
- `sync-job.ts` `cloneToTarget()`: fingerprint dedup + orphan cleanup on dup mapping failure
