# 2026-05-05

## More ghost events — windowed cleanup added

User reported the ghost-event problem returned for VSX calendar (38 ghosts found this time, including weekly meetings: Katrin, Benny, Christoph-Patrick, Olga, Matthias, etc). The April 29 fix only triggered orphan cleanup on full sync, which means ghosts persisted for any source where delta sync was running cleanly but had silently dropped deletions.

Diagnosis: Outlook delta sync occasionally misses deletions (presumably due to delta-token edge cases — events deleted outside the delta window, or reschedule+delete in the same session). The bug isn't in our code — it's a known Outlook Graph API quirk.

Resolution today:
1. Cleared syncToken on the VSX source → triggered manual full sync via BullMQ → orphan cleanup removed 38 ghosts.
2. Added **windowed orphan cleanup**: every 6h, after each successful sync, fetch event IDs in the next 7 days from the provider and remove any DB events not in that set. Runs as fire-and-forget side-channel to delta sync.

Decisions:
- **7-day window only**: User explicitly requested this — frequent cleanup of the visible booking range rather than expensive full-window scans.
- **6h interval**: Balances freshness vs API costs. Tracks `lastWindowedCleanupAt` per source.
- **Optional provider method**: `listEventIdsInRange()` returns just IDs — minimal payload for orphan check. Apple/CalDAV not implemented (delta sync via RFC 6578 sync-tokens behaves differently).
- **Cleanup after success only**: If sync errored, skip cleanup. Avoids deleting events when the provider is misbehaving.

Forward-looking: if Outlook ghost events keep recurring even with windowed cleanup, the next escalation is to abandon delta sync entirely and use windowed full sync as the primary mechanism.
