# 2026-03-15

## Performance optimization — DB indexes, batch sync, and session refetch loop fix

Investigated and fixed multiple layers of performance issues causing 3-5s page loads even locally with one user.

### Changes Made

**Backend:**
- Added 5 DB indexes to `schema.prisma`: `CalendarSource(userId)`, `CalendarEntry(sourceId)`, `Event(calendarEntryId)`, `Event(startTime, endTime)`, `TargetEventMapping(targetCalendarEntryId)` — eliminates full table scans
- Batched sync operations in `sync-job.ts`: created events now upsert in a single `prisma.$transaction()`, deletions use batch `findMany` + `deleteMany`, target event updates run with `Promise.all` in chunks of 5
- Removed worker rate limiter in `worker.ts` (`max: 1, duration: 1000`) that was bottlenecking BullMQ's `concurrency: 5` setting
- Added `@fastify/compress` to `server.ts` for API response compression in dev
- Added nginx gzip compression in `default.conf` for JSON/CSS/JS responses
- Disabled Prisma dev query logging in `prisma.ts` (was logging every SQL to stdout)

**Frontend:**
- Fixed the **main performance bug** in `lib/auth.ts`: `createSigner()` was called in the NextAuth `session` callback (runs on every session read), producing a new JWT each time due to changing `iat`. Moved token signing to the `jwt` callback so it's done once at login and cached.
- Fixed `CalendarView.tsx`: extracted stable `accessToken` as effect dep instead of `session` object, wrapped filtered events in `useMemo`, extracted `eventContent` into `useCallback`, added date equality check in `datesSet` to prevent unnecessary state updates
- Fixed `CalendarSidebar.tsx`: same `session` -> `accessToken` dependency fix
- Fixed `settings/page.tsx`: same fix
- Set `refetchOnWindowFocus={false}` and `refetchInterval={5 * 60}` on `SessionProvider` in `providers.tsx`

### Verification
- TypeScript compiles clean (both packages)
- DB indexes applied via `prisma db push` inside container — EXPLAIN ANALYZE shows 1ms query time for month view (12.5k events, 139 returned)
- Gzip confirmed working via `Content-Encoding: gzip` header
- All Docker services healthy after restart

### Notes
- **Root cause discovery**: The 3-5s delay was NOT the database (query was already 1ms with indexes). It was a **session refetch loop**: NextAuth's `SessionProvider` polls the session, each poll generated a new JWT (different `iat`), which changed `accessToken`, which triggered `useEffect` hooks in CalendarSidebar/CalendarView/SettingsPage to re-fetch all API data, causing a cascade of redundant requests.
- Existing user sessions won't have the cached `accessToken` in their JWT — users need to sign out and back in after this deploy.
- The `@fastify/compress` package needs to be installed inside the Docker container (node_modules aren't bind-mounted). The production Dockerfile handles this via `npm install` during build.

**OpenSpec changes created:**

## Proton Calendar setup page, admin UI, CI/production fixes

Built Proton Calendar integration, full admin UI with three pages, fixed CI pipeline, and resolved production OAuth login.

### Changes Made

**Backend:**
- Deleted `providers/proton.ts` (dead CalDAV code), removed `PROTON` from Provider enum and `getProvider()`
- Added `role` to JWT claims — `AuthUser` interface, `authenticate()` middleware, login/OAuth endpoints all return `role`
- Created `requireAdmin` middleware in `lib/auth.ts` — checks `user.role === "admin"`, returns 403 otherwise
- Admin user seeding in `server.ts` — upserts `admin@admin.local` / `adminpass123` with role `admin` on startup
- New `routes/admin.ts` with three endpoints:
  - `GET /api/admin/users` — paginated, searchable user list with calendar count and OAuth providers
  - `GET /api/admin/sync` — BullMQ job counts + recent jobs with resolved user email and source label, searchable/paginated
  - `GET /api/admin/sources` + `PATCH /api/admin/sources/:id` — list all sources, update sync interval per source (reschedules BullMQ job)
- Sync dedup: `jobId: sync-{sourceId}` on manual triggers + `activeSyncs` Set guard in worker to prevent concurrent syncs per source

**Frontend:**
- New `/settings/proton/page.tsx` — guided Proton Calendar setup with step-by-step instructions, URL + label inputs, calls existing `POST /api/ics/subscribe`
- New `/admin/page.tsx` — searchable users table with email, name, providers (chips), calendar count, join date, pagination
- New `/admin/sources/page.tsx` — all sources with sync interval dropdown (1min–24h), status, last sync, user email
- New `/admin/sync/page.tsx` — job count summary cards, searchable jobs table, click-to-expand error modal
- `AdminSidebar` component — replaces CalendarSidebar on admin pages with Users/Sources/Sync Queue nav
- `AppShell` accepts optional `sidebar` prop for custom sidebar content
- Added `/admin/:path*` to NextAuth middleware matcher
- Admin nav link in AppShell header (visible only to admin role)
- Session now exposes `role` for client-side admin checks

**CI/CD:**
- Added `prisma generate` step before typecheck in `.github/workflows/ci.yml`
- Added `NEXTAUTH_SECRET` + `NEXTAUTH_URL` env vars to turbo build and Dockerfile
- Fixed web Dockerfile: standalone output uses flat structure (not monorepo-nested), copied `node_modules` into `packages/web/` for nft trace, updated CMD to `node server.js`
- Fixed implicit `any` types in `admin.ts` and `target-calendar.ts` map callbacks
- Added env passthrough in `turbo.json` for `NEXTAUTH_SECRET` and `NEXTAUTH_URL`

**Infrastructure (k8s-infra):**
- Added `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `MICROSOFT_CLIENT_ID`, `MICROSOFT_CLIENT_SECRET`, `MICROSOFT_TENANT_ID` env vars to web deployment in `web.yaml`
- Updated `secrets-template.yaml` with all required secret keys for both web (NextAuth) and api (calendar providers)

### Verification

- Admin UI verified in browser via Chrome DevTools MCP: users page, sync queue, sources page all render correctly
- Google OAuth login verified working locally and on production after k8s secret + deployment update

### Notes
- **Production OAuth was broken** because the web pod only had 4 env vars (NEXTAUTH_URL, NEXTAUTH_SECRET, NEXT_PUBLIC_API_URL, API_URL). The api pod uses `envFrom` which injects everything, but the web pod used individual `env` entries and OAuth vars were never listed.
- **Web Docker standalone issue**: Next.js 15.5 standalone output in a monorepo puts `server.js` at root, not under `packages/web/`. The previous Dockerfile assumed the nested path. Fix: copy `node_modules` into `packages/web/` for nft trace, update paths in runner stage.
- Admin user uses `admin@admin.local` (not `admin`) because the sign-in form has `type="email"` validation.
- Sync interval changes via admin UI immediately reschedule the BullMQ job via `queue.upsertJobScheduler()`.
- Memory saved: no sync status indicator needed in the UI (user feedback).
