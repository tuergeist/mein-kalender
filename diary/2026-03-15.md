# 2026-03-15

## Performance optimization — DB indexes, batch sync, and session refetch loop fix

Investigated and fixed multiple layers of performance issues causing 3-5s page loads even locally with one user.

### Changes Made

**Backend:**
- Added 5 DB indexes to `schema.prisma`: `CalendarSource(userId)`, `CalendarEntry(sourceId)`, `Event(calendarEntryId)`, `Event(startTime, endTime)`, `TargetEventMapping(targetCalendarEntryId)` — eliminates full table scans
- Batched sync operations in `sync-job.ts`: created events now upsert in a single `prisma.$transaction()`, deletions use batch `findMany` + `deleteMany`, target event updates run with `Promise.all` in chunks of 5
- Removed worker rate limiter in `worker.ts` (`max: 1, duration: 1000`) that was bottlenecking BullMQ's `concurrency: 5` setting
- Added `@fastify/compress` to `server.ts` for API response compression in dev
- Added nginx gzip compression in `default.conf` for JSON/CSS/JS responses
- Disabled Prisma dev query logging in `prisma.ts` (was logging every SQL to stdout)

**Frontend:**
- Fixed the **main performance bug** in `lib/auth.ts`: `createSigner()` was called in the NextAuth `session` callback (runs on every session read), producing a new JWT each time due to changing `iat`. Moved token signing to the `jwt` callback so it's done once at login and cached.
- Fixed `CalendarView.tsx`: extracted stable `accessToken` as effect dep instead of `session` object, wrapped filtered events in `useMemo`, extracted `eventContent` into `useCallback`, added date equality check in `datesSet` to prevent unnecessary state updates
- Fixed `CalendarSidebar.tsx`: same `session` -> `accessToken` dependency fix
- Fixed `settings/page.tsx`: same fix
- Set `refetchOnWindowFocus={false}` and `refetchInterval={5 * 60}` on `SessionProvider` in `providers.tsx`

### Verification
- TypeScript compiles clean (both packages)
- DB indexes applied via `prisma db push` inside container — EXPLAIN ANALYZE shows 1ms query time for month view (12.5k events, 139 returned)
- Gzip confirmed working via `Content-Encoding: gzip` header
- All Docker services healthy after restart

### Notes
- **Root cause discovery**: The 3-5s delay was NOT the database (query was already 1ms with indexes). It was a **session refetch loop**: NextAuth's `SessionProvider` polls the session, each poll generated a new JWT (different `iat`), which changed `accessToken`, which triggered `useEffect` hooks in CalendarSidebar/CalendarView/SettingsPage to re-fetch all API data, causing a cascade of redundant requests.
- Existing user sessions won't have the cached `accessToken` in their JWT — users need to sign out and back in after this deploy.
- The `@fastify/compress` package needs to be installed inside the Docker container (node_modules aren't bind-mounted). The production Dockerfile handles this via `npm install` during build.

## CI fixes, target sync lifecycle, sidebar persistence, UI fixes

Fixed CI pipeline (3 issues) and implemented several features.

### Changes Made

**CI/CD:**
- Updated `.github/workflows/ci.yml` for merged backend package structure (old refs to `packages/api/`, `packages/sync-worker/`, `@calendar-sync/shared`)
- Added missing `typescript-eslint` and `@eslint/js` root deps, pinned `@eslint/js` to v9 to match eslint 9.x
- Added `@types/node` to web package for Docker build
- Added explicit types on sync-job map callbacks for strict mode

**Target sync lifecycle (`sync-job.ts`):**
- Deletion propagation: `syncCalendarEntry()` now resolves target provider context and calls `deleteEvent()` on the target before removing local events+mappings
- Orphan cleanup: `cloneToTarget()` starts by finding TargetEventMappings whose source events no longer exist, deletes them from target provider
- Change-aware updates: added `lastSyncedAt` to TargetEventMapping, update loop now skips events where `sourceEvent.updatedAt <= mapping.lastSyncedAt`
- Duplicate prevention: mapping creation wrapped in try-catch for unique constraint violations

**Sidebar calendar toggle persistence:**
- Added `PATCH /api/calendar-entries/:id` endpoint to toggle `enabled` field
- Sidebar now initializes from `entry.enabled` (API state) instead of showing all calendars
- Toggle does optimistic UI update then persists via API
- Removed separate `visibleCalendars` state — uses `entry.enabled` directly

**UI:**
- Fixed working location events showing only emoji icon without title text
- Fixed HeroUI Select for target calendar (changed to Set-based selectedKeys, items prop, disallowEmptySelection)

**OpenSpec changes created:**
- `proton-cal-integration` — Proton Calendar via ICS share link (not CalDAV), all artifacts complete
- `target-sync-cleanup` — target sync lifecycle, all artifacts complete + implemented

### Notes
- Proton Calendar has no external write API — only ICS share links (read-only). The existing `providers/proton.ts` CalDAV implementation is dead code. Integration will reuse existing ICS subscription backend.
- CI deploy step fails on missing `K8S_INFRA_PAT` secret — repo settings issue, not code. All build jobs pass green.
- The `cloneToTarget()` function resolves target credentials independently from the source provider — needed because deletion propagation happens in `syncCalendarEntry()` which only has source context.
