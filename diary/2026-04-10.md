# 2026-04-10

## Proton Calendar booking via ICS email invitations

Implemented the full stack for enabling Proton Calendar (and any ICS-only calendar) as a booking target. Since Proton has no write API, we send ICS email invitations (RFC 5546 iTIP) from noreply@mein-kalender.link.

**Architecture decision:** No new CalendarProvider — instead, the booking flow checks `bookingEntry.source.emailForInvitations`. If set, sends METHOD:REQUEST/CANCEL ICS via email. If not, uses existing Google/Outlook API path. Clean branching, no abstraction overhead.

**Key new modules:**
- `lib/ics-utils.ts` — extracted shared ICS formatting from ics-serve.ts
- `lib/ics-invitation.ts` — RFC 5545 VCALENDAR builder with METHOD:REQUEST/CANCEL, SEQUENCE tracking, ORGANIZER/ATTENDEE
- `lib/email.ts` — nodemailer with lazy init (like Mollie), won't crash without SMTP config
- Email queue (BullMQ, concurrency 3) — fire-and-forget, doesn't block booking response

**Schema:** `CalendarSource.emailForInvitations`, `Booking.icsUid` + `icsSequence` for lifecycle tracking across create/reschedule/cancel.

**Reschedule:** Sends updated METHOD:REQUEST with incremented SEQUENCE (same UID). Cancel sends METHOD:CANCEL. Both to host AND guest.

## Sync token reset on failure

Found that "fetch failed" errors during Google sync caused events to be lost — the delta sync token advanced past events created during the outage. Fix: reset syncToken to null on any sync error, forcing full fetch on next run.

## Clone sync investigation

"Sync M-C" event reported missing. Investigation showed it never appeared in our DB despite Google source showing syncStatus:ok. Root cause: intermittent "fetch failed" errors on 9. Apr around the time the event was created. The sync token had already advanced past it. Full token reset recovered other events but this specific one may have had a different name in Google Calendar.
