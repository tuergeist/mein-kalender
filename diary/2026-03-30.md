# 2026-03-30

## Outlook "(No title)" Bug — Root Cause + Fix

### Was passiert ist

Outlook-Termine im Calendar View und Dashboard zeigten "(No title)" oder leere Titel statt der echten Betreffzeilen. Nur einzelne Events (z.B. "Allignment IT") hatten ihren Titel behalten.

### Root Cause

Microsofts Graph API hat zwei Endpunkte fuer Kalender-Events:

1. **`calendarView`** — gibt komplette Events mit allen Feldern zurueck
2. **`calendarView/delta`** — gibt nur geaenderte Felder zurueck (sparse objects)

Unser Sync nutzte `calendarView/delta` fuer BEIDES: initialen Full Sync UND inkrementelle Updates. Bei recurring Events (woechentliche Meetings etc.) liefert der Delta-Endpunkt oft nur `start`, `end` und `type` — aber kein `subject`. Unser Code interpretierte fehlendes `subject` als leeren String und schrieb das in die DB.

Drei Fehler lagen uebereinander:

1. **Full Sync nutzte den falschen Endpunkt.** `calendarView/delta` statt `calendarView` bedeutete, dass selbst ein kompletter Resync sparse Daten lieferte.

2. **Der upsert-Update-Pfad ueberschrieb blind.** Wenn ein Event schon in der DB existierte und der Sync ein Update brachte, wurde der Titel mit leerem String ueberschrieben, auch wenn der Provider das Feld gar nicht geliefert hatte.

3. **Keine Unterscheidung zwischen "Feld fehlt" und "Feld ist leer".** Der Code behandelte `subject: undefined` (nicht im Response) gleich wie `subject: ""` (echtes Event ohne Titel).

### Was gefixt wurde

1. **`outlook.ts`: Full Sync nutzt jetzt `calendarView` (ohne `/delta`)** — gibt immer komplette Events mit Subject zurueck. Der Delta-Endpunkt wird nur noch fuer inkrementelle Syncs mit bestehendem syncToken genutzt. Nach dem Full Fetch wird separat ein deltaLink geholt fuer zukuenftige Updates.

2. **`outlook.ts`: `mapEvent` unterscheidet jetzt drei Faelle:**
   - `subject` vorhanden mit Wert → normaler Titel
   - `subject` vorhanden aber null/leer → "(No title)" (echtes titelloses Event)
   - `subject` fehlt im Response → setzt `_subjectMissing` Flag (Delta hat das Feld nicht geliefert)

3. **`sync-job.ts`: Beide Pfade (created/upsert UND updated) respektieren die `_subjectMissing` Flag** — wenn der Provider das Subject nicht geliefert hat, wird der bestehende Titel in der DB nicht ueberschrieben.

### Warum es nicht wieder vorkommen wird

- Full Sync und Delta Sync nutzen jetzt **verschiedene API-Endpunkte**. Es ist strukturell unmoeglich, dass ein Full Sync sparse Daten bekommt.
- Der Delta-Sync-Pfad kann bestehende Titel nicht mehr ueberschreiben, wenn das Subject nicht im Response war. Das Flag-System (`_subjectMissing`) ist explizit, nicht implizit.
- Die gleiche Logik gilt fuer `body` und `location` (`_bodyMissing`, `_locationMissing`).

### Verifikation

Nach dem Fix: syncToken geloescht, Full Resync getriggert. 540 Events total, alle heutigen Events haben korrekte Titel (Katrin weekly, Management Meeting, Olga weekly, Lunch, privat, Fokuszeit). 51 Events in der Zukunft (Juni) zeigen "(No title)" — das sind echte titellose Events (automatische Blocker/Fokuszeiten von Outlook).
